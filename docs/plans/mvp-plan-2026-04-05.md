# MVP Plan: Desktop Voice Orchestrator for tmux Agent Sessions

**Date:** 2026-04-05
**Status:** Proposed
**Related Docs:** [../research/initial-research-2026-04-05.md](../research/initial-research-2026-04-05.md)

## Outcome

Create a local service that lets the user talk to one orchestrator agent session by voice while that orchestrator can inspect and coordinate other tmux-hosted Claude Code or Codex sessions.

## Scope

In scope for MVP:

- session and pane discovery across the current tmux server
- pane tagging and aliasing
- recent-output reads from panes
- text injection into a chosen pane
- copy output from one pane and paste it into another pane
- a push-to-talk voice conversation loop with one designated orchestrator pane
- a mobile-friendly web client for recording, transcript display, and reply playback
- a swappable orchestrator harness boundary with Claude Code CLI first
- a swappable TTS boundary with Cartesia first

Out of scope for MVP:

- custom wake-word hardware behavior
- Raspberry Pi deployment
- full autonomous multi-agent scheduling
- broad GUI automation beyond tmux panes
- native mobile app as the first client

## Proposed Stack

### Core daemon

- TypeScript on Node
- one long-lived local process
- JSON API plus streamed event channel

Rationale:

- aligns with the reusable Halo-2 harness patterns
- strong support for child processes, tmux control clients, and local HTTP/SSE
- easier to share conventions with the existing `pi-agent` bridge code without adopting `pi-coding-agent`

### Orchestrator harness

- primary implementation: Claude Code CLI wrapper
- fallback implementation: pi-coding-agent
- both implementations must satisfy the same local interface

Rationale:

- Claude Code CLI is already installed and locally verified to stream structured output in non-interactive mode
- keeps the user's current Claude Code workflow and skills/plugins model closer to intact
- leaves room to switch later if CLI constraints become painful

### Voice layer

- abstract speech adapter interface from day one
- first implementation should be explicit push-to-talk with `start` and `stop`
- future implementation can reuse Pipecat once the tmux control plane is stable

### TTS

- abstract TTS adapter interface from day one
- first implementation: Cartesia streaming TTS
- later options: ElevenLabs, Gemini TTS, or local engines

### Client

- responsive web app first
- optimized for phone use from the start
- backend contract kept clean so a native mobile app can be added later without reworking the server

## Core Components

### 1. Session registry

Responsibilities:

- discover tmux sessions, windows, and panes
- store aliases and role labels
- remember which pane is the `orchestrator`
- expose a stable targeting model independent of raw session names

Suggested data:

- pane ID
- session name
- window name
- pane title
- current command
- role
- alias
- capture settings

### 2. tmux observer

Responsibilities:

- open control-mode clients for selected sessions
- parse `%output` and lifecycle notifications
- maintain recent scrollback snapshots
- resync with `capture-pane` when needed

Key design choice:

- hybrid model instead of polling-only
- use control mode for hot sessions
- use `capture-pane` for cold sessions, resync, and alternate-screen fallback

### 3. tmux actuator

Responsibilities:

- send literal text into panes
- send special keys like `Enter`, `Escape`, or `C-c`
- load tmux buffers and paste them into target panes
- perform explicit source-to-target copy operations

Recommended primitives:

- `send-keys -l`
- `send-keys Enter`
- `load-buffer -`
- `paste-buffer -p -r`

### 4. Orchestrator bridge

Responsibilities:

- accept transcribed user speech from the client
- pass it to the selected harness
- watch orchestrator output
- decide when a reply is complete enough to speak

MVP completion heuristic:

- stream text as it appears
- consider a turn complete after output silence plus prompt reappearance
- keep the heuristic configurable per pane type

### 5. Voice adapter

Responsibilities:

- record until the user stops speaking
- return audio or transcription to the backend
- speak streamed or chunked reply text
- support interruption and cancellation

The tmux orchestration logic must not depend on one specific speech backend.

### 6. TTS adapter

Responsibilities:

- accept streaming assistant text chunks
- buffer text into provider-appropriate units if needed
- emit playable audio chunks or files
- support provider swaps without changing client code

### 7. Web client

Responsibilities:

- show a large start/stop speaking control
- show transcript and orchestrator reply text
- play reply audio
- show connection state and the active orchestrator target
- work well on desktop and mobile browsers

## Execution Sequence

### Milestone 1: tmux core

- implement discovery and registry
- implement pane snapshot reads
- implement safe text injection
- implement explicit copy-from-pane to target-pane paste flow

### Milestone 2: harness wrapper

- implement a Claude Code CLI runner
- parse `stream-json` events into a normalized internal event stream
- keep the interface narrow enough to support a future `pi-coding-agent` implementation

### Milestone 3: live observation

- add control-mode client management
- parse `%output` notifications
- attach per-session observers for the sessions marked active
- add fallback resync with `capture-pane`

### Milestone 4: orchestrator session contract

- define how the orchestrator pane is identified
- add reply-complete heuristics
- add event logging for user turn, pane injection, pane output, and cross-pane copy actions

### Milestone 5: web voice client

- add a simple browser client
- wire microphone start/stop actions
- send audio or transcription to the backend
- play reply audio or TTS output

### Milestone 6: streaming TTS

- add a Cartesia-backed TTS adapter on the backend
- connect normalized harness text chunks to TTS streaming
- support interruption and cancellation

### Milestone 7: voice loop hardening

- add a simple speech adapter on the backend
- harden text chunk aggregation for TTS quality
- add provider swap seams for TTS

### Milestone 8: hardening

- add allowlists or role-based restrictions for writable panes
- add rate limits and timeout handling
- add a dry-run or preview mode for cross-pane paste actions
- add auth plus secure remote access assumptions for phone use away from home

## Immediate Next Build Step

Start with the tmux core, not the mobile app.

The first code slice should implement:

1. `list sessions and panes`
2. `capture recent output from a pane`
3. `inject text into a pane`
4. `copy output from pane A into pane B using tmux buffers`
5. `store aliases and the designated orchestrator pane in local state`

After that, add a minimal web client with:

1. `start speaking`
2. `stop speaking`
3. `transcript display`
4. `reply playback`

Immediately after the tmux core, implement the Claude Code CLI wrapper before building out the client UI.
